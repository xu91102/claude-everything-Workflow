# 架构原则

## 核心原则 (CRITICAL)

| 原则 | 说明 |
|------|------|
| **单一职责 (SRP)** | 每个模块/类/函数只做一件事 |
| **开闭原则 (OCP)** | 对扩展开放，对修改关闭 |
| **依赖倒置 (DIP)** | 依赖抽象而非具体实现 |
| **低耦合高内聚** | 模块间依赖最小化，模块内功能聚焦 |
| **DRY** | 避免重复代码，但不过度抽象 |
| **KISS** | 保持简单，避免过度设计 |

## 目录组织原则

```
按业务/功能模块组织目录
├── 按领域划分顶层目录
├── 相关文件放在一起
├── 公共代码抽离到 common/shared/utils
└── 配置文件放项目根目录或 config 目录
```

## 分层架构（后端项目）

```
表现层 (Controllers) → 请求/响应处理
    ↓
业务层 (Service) → 业务逻辑
    ↓
数据层 (Repository) → 数据访问
    ↓
模型层 (Model/Entity) → 数据结构
```

## 前端分层架构

**核心原则**：UI 与逻辑分离

| 框架 | 逻辑抽离 | 命名规范 |
|------|----------|----------|
| Vue | Composables | `useXxx.ts` |
| React | Custom Hooks | `useXxx.ts` |

**目录结构**：

```
views/        → 页面组件，仅布局和组合
components/   → UI 组件，纯展示
composables/  → 业务逻辑封装
api/          → 接口请求
stores/       → 状态管理
types/        → 类型定义
```
