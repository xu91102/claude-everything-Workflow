# 性能优化

## 模型选择策略

| 模型   | 适用场景                           | 成本 |
| ------ | ---------------------------------- | ---- |
| Haiku  | 轻量代理、高频调用、Worker 代理    | 低   |
| Sonnet | 主要开发工作、多代理编排、复杂编码 | 中   |
| Opus   | 深度架构推理、复杂分析、研究任务   | 高   |

**默认使用 Sonnet**，仅在需要深度推理时切换 Opus。

### 上下文窗口管理

- MCP 工具描述消耗 Token，控制同时启用的 MCP 数量 (<10)
- 控制活跃工具数 (<80)
- 使用 `disabledMcpServers` 禁用未使用的 MCP

### 上下文压缩规则

#### 建议压缩

- 研究/探索完成后，准备进入实施前。
- 一个里程碑完成后，准备开始下一个阶段前。
- 调试完成后，准备回到正常开发节奏前。
- 明确放弃失败方案后，准备切换到新方案前。
- 会话已积累大量无关历史，且接下来的工作只依赖当前结论、关键文件和下一步动作时。

#### 禁止压缩

- 正在实施跨多文件功能时。
- 正在调试复杂问题，仍依赖最近的错误日志、变量名、调用链或临时假设时。
- 还未完成当前阶段的结论整理时。
- 刚完成代码阅读，但尚未把关键发现、影响文件和下一步动作明确下来时。
- 还未确认用户最新约束、验收标准或风险边界时。

#### 压缩前必须完成

- 用简短文字总结当前目标。
- 记录已完成项和未完成项。
- 记录关键文件、关键接口和关键决策。
- 记录剩余风险、阻塞点和下一步动作。
- 如果存在待验证假设，明确标出，避免压缩后误当成已确认事实。

#### 压缩后必须确认

- 当前目标是否仍然一致。
- 下一步动作是否清晰。
- 是否需要重新打开关键文件恢复精确上下文。
- 是否有因为压缩而丢失的约束、路径、变量名或错误细节。

## 代理团队成本

Agent Teams 会产生多个独立上下文窗口。
仅在并行处理有明确价值时使用（多模块工作、并行审查）。
简单顺序任务使用子代理更省 Token。

## 上下文窗口低区域

避免在上下文窗口的最后 20% 执行:

- 大规模重构
- 跨多文件的功能实现
- 复杂交互调试

低上下文敏感度任务:

- 单文件编辑
- 独立工具类创建
- 文档更新
- 简单 Bug 修复
